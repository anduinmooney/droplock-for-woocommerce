# v1.1 — Per-variation limits

Design spec for the v1.1 milestone of DropLock for WooCommerce (Pro). The goal: let store owners set a separate limit per variation of a variable product, instead of one shared limit on the parent.

This is the **#1 requested upgrade** in the existing feature backlog and the natural next step after v1.0.

---

## Goal

> A store sells a "Limited Edition Tee" with three variations: Red, Blue, Green. The store owner wants each color to have its own per-customer limit — say, Red = 1, Blue = 2, Green = unlimited. v1.0 only supports a shared parent-level limit. v1.1 makes per-variation limits possible.

---

## Scope

### In scope

- A new per-variation toggle: **"Override parent DropLock setting for this variation."**
- When overridden: per-variation `max_qty`, custom message, and badge text.
- Validation logic queries the relevant scope: parent if no override exists, variation if override exists.
- Migration: existing v1.0 customers' settings continue working unchanged (parent-only) — opt-in behavior change.
- Admin UI in the variation editor (the per-variation accordion under the Variations tab).

### Out of scope (still parent-level in v1.1)

- The product-page badge (still derived from parent in v1.1).
- The blocked-attempt log columns (still log parent product, not variation).
- "Total across all variations" (a separate parent-level limit that ALSO caps the sum) — that's a v1.2 thing if customers ask.
- The customer-debug panel (separate v1.1 ship).

---

## Data model

### New product/variation meta

| Key | On | Type | Meaning |
|---|---|---|---|
| `_droplock_variation_override` | Variation post | `'yes'` / `'no'` | "Use my own setting, not the parent's" |
| `_droplock_max_qty` | Variation post | int | Same key as parent — for variations that override |
| `_droplock_limit_message` | Variation post | string | Same key as parent — variations that override get their own message |
| `_droplock_badge_text` | Variation post | string | Optional in v1.1 — variation badge is a v1.2 stretch |

The `META_ENABLED` and `META_ORDER_STATUSES` keys stay parent-only. Sub-statuses don't change per variation — that'd be too fine-grained for v1.1.

### No DB schema changes

All data lives in `postmeta`. Variations are already a CPT (`product_variation`) with their own postmeta. The blocked-attempt table stays unchanged in v1.1.

## Admin UI

### On the variable product → Variations tab

Each variation accordion (which already has Regular Price, SKU, etc.) gets a new fieldset:

```
[ DropLock override ]
  [ ] Override parent DropLock setting for this variation

  ─── (when checked, the fields below appear) ───
  Maximum quantity per this variation: [   1   ]
  Custom limit message (optional):      [           ]
```

The override checkbox is the only field shown when collapsed. Saving cascades to standard `woocommerce_save_product_variation` flow.

### On the parent product → DropLock section

A new informational line:

> **Variations with overrides:** 2 of 3 use per-variation limits ([Red](#), [Blue](#)). [Green](#) inherits this parent limit.

This is read-only — admins manage overrides on each variation accordion, not centrally. (Bulk variation editing arrives in v1.2.)

## Validation logic — the core change

### Current v1.0 behavior

```
get_effective_product_id($product_id):
    if it's a variation → return parent_id
    else → return product_id

is_enabled($effective_id)   ← checks parent meta
get_max_qty($effective_id)  ← checks parent meta
```

### New v1.1 behavior

Introduce a `get_limit_context($product_id, $variation_id)` that returns the right ID and scope:

```php
function get_limit_context( $product_id, $variation_id ) {
    $parent_id = $variation_id
        ? wc_get_product( $variation_id )->get_parent_id()
        : $product_id;

    // If the variation explicitly overrides, return variation scope.
    if ( $variation_id ) {
        $override = get_post_meta( $variation_id, '_droplock_variation_override', true );
        if ( 'yes' === $override ) {
            return [
                'enabled_id'    => $parent_id,         // parent still owns the on/off
                'limit_id'      => $variation_id,      // variation owns the max qty + message
                'product_match' => 'variation',        // count only orders of THIS variation
                'parent_id'     => $parent_id,
            ];
        }
    }

    // Default — parent-scoped.
    return [
        'enabled_id'    => $parent_id,
        'limit_id'      => $parent_id,
        'product_match' => 'parent',                    // count all variations rolled up
        'parent_id'     => $parent_id,
    ];
}
```

Then everywhere validation runs, instead of `$effective_id` it uses `$ctx = get_limit_context(...)` and:

- `DropLock_Helper::is_enabled( $ctx['enabled_id'] )`
- `DropLock_Helper::get_max_qty( $ctx['limit_id'] )`
- `DropLock_Helper::get_limit_message( $ctx['limit_id'] )`

### Order-history query change

`DropLock_Order_Query::get_customer_purchased_quantity` currently matches a line item if `product_id == parent` OR `variation_id rolls up to parent`. v1.1 adds a `product_match` param:

```php
get_customer_purchased_quantity(
    $product_id,                // parent id either way
    $user_id, $billing_email, $statuses,
    $product_match = 'parent',  // 'parent' (default) or 'variation:<id>'
)
```

When `$product_match === 'variation:42'`, count only line items where `$item->get_variation_id() === 42`. Otherwise, current rollup behavior.

This is a minor change — one extra branch in the inner loop. Cache key includes the match scope so parent-scope and variation-scope lookups don't clobber each other.

## Migration story for existing v1.0 customers

**Goal: zero-change behavior on upgrade.**

- All existing products have `_droplock_variation_override` absent (= empty / not 'yes') on every variation.
- `get_limit_context()` therefore returns the parent-scoped result everywhere.
- Customers see no behavioral change until they explicitly check "Override" on a variation.

**No DB migration needed.** No `dbDelta`, no admin notice, no on-upgrade hook. Just ship the new code.

**Version constant bumps:** `DROPLOCK_VERSION` → `1.1.0`, `DROPLOCK_PRO_VERSION` → `1.1.0`. No `DROPLOCK_DB_VERSION` bump (DB unchanged).

## Edge cases to handle

| Case | Behavior |
|---|---|
| Variation has override, parent has DropLock disabled | Variation override is ignored (parent gate wins). Document this. |
| Two variations have different overrides, customer buys one of each | Each variation tracked independently. Buying Red (override=1) doesn't count toward Blue's allowance. |
| Variation override exists, then admin uncheck the override | Save flow deletes `_droplock_variation_override`. Variation reverts to parent-scoped limits. Test that previously-blocked customers can now buy again. |
| Customer bought one variation under v1.0 (parent rollup), admin upgrades to v1.1 and enables variation override | v1.1 should NOT retroactively re-count prior orders against the new per-variation limit. The customer's existing purchase is in their history — we just match it against the new scope. Result: previously bought 1 Red, parent limit was 1, customer was blocked from Blue. Admin now enables override on Red with limit=1: customer is still blocked from Red. Override on Blue with limit=1: customer is now ALLOWED to buy Blue. **This is the desired behavior.** Document it. |
| Custom message templates `{product_name}` on a variation | Use `wc_get_product( $variation_id )->get_name()` which already returns the variation's full name (e.g., "Limited Edition Tee — Red"). |
| Variation is a "subscription variation" (WooCommerce Subscriptions) | Untested in v1.1. Document as "should work, file an issue if it doesn't." |

## Testing checklist

In addition to the v1.0 test suite (re-run all 23):

- [ ] Variable product with 3 variations, override on 1, parent limit on 2: customer buys override variation 1x → blocked from re-buying it. Customer still buys parent-scoped variation up to parent limit.
- [ ] Variable product, override on all 3 variations with different limits: customer can buy each variation up to its own limit, independently.
- [ ] Variable product with override removed from a variation: previously-blocked customer is now subject to parent rule (which may or may not block them).
- [ ] Existing v1.0 product upgraded to v1.1 with no admin action: behavior identical to v1.0.
- [ ] Variation override with custom message: message renders with correct variation name.
- [ ] Variation override with limit 0 (or negative): treated as 1, like parent.
- [ ] Bulk variation generation (WC's "Create variations from all attributes"): newly generated variations have NO override (default empty).
- [ ] Switching a product from variable to simple after configuring variation overrides: overrides become stale meta on (now-orphaned) variation posts. WC handles orphan cleanup; we don't need to.

## Risks

| Risk | Mitigation |
|---|---|
| Order-history query gets slower (separate cache key per variation scope) | Cache key includes match scope; 60s TTL unchanged. Stores with 100k+ orders should still be fine. |
| Admin UI clutter on the variation accordion | Hide all fields behind the override checkbox. Default-collapsed. |
| Existing snippets that filter `droplock_*` meta keys break | Filter keys are unchanged. New behavior is in new code paths. |
| Subscriptions / Bundles interactions | Out of scope. Document, file in issue tracker. |
